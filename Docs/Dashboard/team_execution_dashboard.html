3. Overview

MVP Goals & Critical Path

Nền tảng Volunteer Connect hướng tới số hóa quy trình kết nối Tình nguyện viên (Volunteer) và Nhà tổ chức (Organizer) thông qua sự phê duyệt của Admin và cơ chế điểm danh hoàn thành tích lũy. Dự án tận dụng tối đa cơ chế phi chuẩn hóa (Denormalization) của MongoDB để hạn chế `$lookup` và thực thi bảo vệ ACID qua Replica Set Transactions.

MVP Scope (Phạm vi MVP)
  • Xác thực số điện thoại bằng OTP SMS (Bypass offline qua console khi phát triển).
  • Nâng cấp vai trò Volunteer → Organizer kèm logic thời gian cooldown.
  • Đăng ký hoạt động với thuật toán Overlap Check (trùng lịch) không dùng `$lookup`.
  • Điểm danh sự kiện tự động cộng tích lũy Joined Activity Count cho Volunteer.
  • Volunteer Feed (Posts) chia sẻ kinh nghiệm cộng đồng.
Critical Milestones (Đường găng dự án)
  • Day 3: API Freeze - Khóa cứng toàn bộ cấu trúc Request/Response và Mock API.
  • Day 8: Feature Freeze - Đóng băng hoàn toàn mã nguồn backend, chuyển sang fix bug.
  • Day 9-10: Integration Phase - Tích hợp hệ thống E2E Frontend và Backend thật.
  • Day 12: Release Candidate (RC) - Deploy bản build Staging kiểm thử khói cuối.

6. Backend Ownership

Auth + Users + Requests Dev 1

Router Prefix: `/api/v1/auth`, `/api/v1/users`, `/api/v1/organizer-requests`
MongoDB Collections: `users`, `organizer_requests`
ACID Transactions: None.
Main Endpoints: Register, Verify OTP, Login, Refresh Token, Get/Update profile, Upgrade Request, Request cooldown.
Độ khó: Medium (Yêu cầu mật mã băm, JWT, check cooldown).
Dependency: None (Core module).

Activities Management Dev 2

Router Prefix: `/api/v1/activities`, `/api/v1/organizer/activities`
MongoDB Collections: `activities`
ACID Transactions: **Transaction 8.4** (Organizer hủy hoạt động - đồng loạt cập nhật trạng thái các đơn đăng ký liên quan sang `Cancelled`).
Main Endpoints: CRUD activity, submit for review, cancel activity, keyword Text Search.
Độ khó: Medium (Lập trình transaction, cron job quét status tự động và MongoDB Text Index).
Dependency: Phụ thuộc vào JWT middleware của Dev 1.

Registrations & Attendance Dev 3

Router Prefix: `/api/v1/registrations`, `/api/v1/attendance`
MongoDB Collections: `registrations`, `activities`, `users`
ACID Transactions:
  • **Transaction 8.1:** Đăng ký mới (Lock, check limit, ghi đơn).
  • **Transaction 8.2:** Volunteer tự hủy đơn / từ chối (Hoàn trả counter, auto-open hoạt động từ Full).
  • **Transaction 8.3:** Điểm danh Completed (Đổi trạng thái đơn, cộng counter Profile User).
Main Endpoints: Register to activity, cancel registration, view registrations list, approve/reject registration, attendance check-in.
Độ khó: High (Thuật toán overlap check trùng lịch, bọc transactions session, chống race-condition tranh chấp).
Dependency: Phụ thuộc vào Auth (Dev 1), Activity schema (Dev 2).

Feed Posts & Admin Dev 4

Router Prefix: `/api/v1/posts`, `/api/v1/admin`
MongoDB Collections: `posts`, `activities`, `organizer_requests`, `users`
ACID Transactions: None (Atomic counter update).
Main Endpoints: Create/Delete post, pagination Feed, filter hashtag, like/share post, Admin approve role request, Admin approve activity, Admin statistics charts.
Độ khó: Medium (MongoDB aggregation stats, cursor pagination).
Dependency: Phụ thuộc vào Auth (Dev 1), Activity & Registration schemas (Dev 2, Dev 3).

10. Git Workflow

Quy tắc phân nhánh, Commit và Pull Request

Chiến lược phân nhánh:
  • Nhánh chính: `develop` (Tích hợp), `main` (Phát hành Production).
  • Tách nhánh độc lập cho từng thành viên:
    • Dev 1: `feature/auth-users-profile`
    • Dev 2: `feature/activities-lifecycle`
    • Dev 3: `feature/registrations-attendance`
    • Dev 4: `feature/posts-admin-integration`
    • FE Dev: `feature/frontend-ui-integration`
Commit Convention (Chuẩn viết commit):

Sử dụng cú pháp: `type(scope): message`. Ví dụ: `feat(auth): add verify-otp endpoint` hoặc `fix(registration): resolve overlap date calculation`.

Quy tắc PR & Code Review Assignees:
  • PR của Dev 1 → Review bởi Dev 2 / Tech Lead.
  • PR của Dev 2 → Review bởi Dev 3.
  • PR của Dev 3 → Review bởi Dev 1.
  • PR của Dev 4 → Review bởi Dev 2.
  • PR của FE Dev → Review bởi Tech Lead / Dev 4.
Quy tắc tránh Conflict:

Lập trình viên bắt buộc chạy `git pull origin develop --rebase` dưới local máy mình để giải quyết conflict trước khi tạo PR lên GitHub. File `main.py` và `shared/models.py` bị khóa cứng, chỉ Tech Lead được chỉnh sửa.

12. Project Structure

Cấu trúc thư mục nguồn của Project

Cây thư mục thực tế được thiết lập tại Project/ theo mô hình hướng tính năng:

Project/ ├── backend/ │ ├── app/ │ │ ├── core/ # Lớp cấu hình dùng chung toàn app │ │ │ ├── config/ (settings.py) │ │ │ ├── database/ (mongodb.py, beanie.py) │ │ │ ├── middleware/ (cors.py) │ │ │ ├── security/ (jwt.py, password.py, permissions.py) │ │ │ ├── exceptions/ (handlers.py, custom_exceptions.py) │ │ │ └── utils/ (datetime.py, pagination.py, object_id.py) │ │ │ │ │ ├── features/ # Chứa 8 modules tính năng đóng gói độc lập │ │ │ ├── auth/ │ │ │ ├── users/ │ │ │ ├── organizer_requests/ │ │ │ ├── activities/ │ │ │ ├── registrations/ │ │ │ ├── attendance/ │ │ │ ├── posts/ │ │ │ └── admin/ │ │ │ │ │ ├── shared/ # Phản hồi json và base models chung │ │ └── main.py │ ├── tests/ (unit/, integration/, e2e/) │ ├── scripts/ (database/, seed/) │ ├── requirements.txt │ └── README.md ├── frontend/ (public/, src/) └── README.md
Chi tiết các tệp tin cấu thành một Feature:
  • `router.py`: Chứa các API routes định tuyến và FastAPI parameters (không chứa logic).
  • `models.py`: Khai báo Collection MongoDB Beanie ODM và các schema index của Collection đó.
  • `schemas.py`: Khai báo Pydantic schemas (DTO) kiểm duyệt kiểu dữ liệu đầu vào/đầu ra.
  • `services.py`: Nơi thực thi logic nghiệp vụ cốt lõi và kiểm tra nghiệp vụ.
  • `repositories.py`: Lớp thực thi tương tác trực tiếp DB để dễ dàng tạo Mock DB khi chạy Unit test.
  • `validators.py`: Logic kiểm duyệt nghiệp vụ bổ sung (như cooldown).
  • `dependencies.py`: FastAPI Depends injection.
  • `constants.py`: Hằng số nội bộ của feature.
